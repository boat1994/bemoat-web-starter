North star

Build reusable guardrails once in bemoat-web-starter, sync them into real projects, validate against real issues, then promote only what improves agent output.

  • Improve once: starter holds reusable rules, prompts, checklists, and workflow docs.
  • Test in reality: project repos prove whether the guardrail improves real screens.
  • Promote carefully: only stable, reusable improvements go back into starter.
  • Human taste stays final: AI can suggest and execute, but cannot promote canonical taste alone.

Key split

Place Responsibility
bemoat-web-starter Reusable guardrails, workflows, prompts, visual QA, accessibility, anti-generic UI rules.
bemoat Project-specific homepage reference pack, brand taste, real implementation issue.
Design Ref CMS Raw conversations, references, extracted design memory, reusable patterns, prompt notes.
Composer 2.5 Routine extraction, classification, checklist generation, Cursor prompt creation.
GPT-5.5 High Red team, scope control, risk review, over-engineering check.

10. Core rules

  • Read issue first: no implementation before understanding current AC.
  • Declare guardrails: list exact guardrails used before coding.
  • Plan before code: write implementation plan before editing.
  • No scope creep: do not add unrelated features or redesigns.
  • Map back: after changes, map each change to AC.
  • Self-check: run relevant checklist and list unmet rules.
  • Commit safely: avoid leaving risky uncommitted states before rest.

11. UI rules

  • Smallest useful skill: choose only relevant UI Skills, not every skill.
  • Anti-generic: avoid template-looking, AI-looking, CMS freelancer vibe.
  • Hierarchy first: improve information hierarchy before decoration.
  • State coverage: include loading, empty, error, success when relevant.
  • Responsive check: test desktop and mobile breakpoints.
  • Motion with purpose: define trigger, moving element, timing feel, reduced motion fallback.

12. Design Ref CMS rules

  • CMS stores memory: raw JSON, extracted notes, patterns, avoid rules.
  • Local extraction: Composer 2.5 extracts outside CMS.
  • Dumb API: ingest endpoint validates, saves, links, returns id.
  • No smart API v1: no deep design reasoning inside API.
  • Parent link: extracted record must link back to raw source.
  • Status discipline: raw, extracted, used, promoted, archived.

13. Promotion rules

  • One round, one capability: do not mix many new systems in one round.
  • Project proof first: test guardrail on one real issue.
  • Promote only reusable: brand-specific taste stays in project repo or CMS.
  • Human approval: agent cannot promote canonical pattern alone.
  • Record deltas: write what improved and what failed.
  • No dump: never turn starter into a reference image dump.

17. Master prompt for agents

Before implementation:
1. Read the current issue and acceptance criteria.
2. Select only the relevant guardrails.
3. List the exact guardrail files used.
4. List what is intentionally out of scope.
5. Write an implementation plan before coding.

During implementation:
1. Keep changes mapped to the issue.
2. Do not introduce unrelated business logic.
3. Do not redesign unrelated sections.
4. Prefer small, reviewable changes.

After implementation:
1. Map each change back to acceptance criteria.
2. Run the relevant visual QA and accessibility checklist.
3. List any unmet guardrail.
4. Write visual delta notes if UI changed.
5. Stop at the smallest complete slice.
